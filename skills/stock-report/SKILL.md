# A股深度研究报告 · 全流程编排

本技能是流水线指挥官，按顺序协调以下技能共同完成分析：

```
[stock-report]          ← 你在这里，负责编排全流程
    ├── research-triangulation   数据来源验证
    ├── analytical-rigor         分析严谨性检查
    ├── council                  四视角独立辩论
    └── HTML生成 + Git推送
```

**所有技能必须安装**（见 analysis-skills 仓库）：
`research-triangulation` · `analytical-rigor` · `council`（ECC内置）

---

## 触发方式

```
/stock-report <股票代码> <股票名称>
/stock-report <股票代码> <股票名称> 持仓<金额> 持仓<天数>天
```

示例：
```
/stock-report 600519 贵州茅台
/stock-report 300750 宁德时代 持仓20万 持仓90天
```

用户说「帮我分析一下600519」「研究下宁德时代」也触发本技能。

## Step 0：配置检查（只做一次）

检查 `~/.claude/skills/stock-report/config.md` 是否存在。

**不存在时**，自动检测仓库路径和 GitHub Pages URL：
1. 检查当前目录是否有 `_DESIGN_SYSTEM.md`
2. 检查当前目录是否有 `index.html` + `.git`
3. 检查主目录下是否有 `kin-g-signal` 文件夹
4. 以上都没有 → 询问用户本地报告目录路径

检测到后，从 git remote 推导 Pages URL：
`https://github.com/user/repo` → `https://user.github.io/repo`

写入配置：
```
repo_path: {路径}
pages_url: {URL}
configured_at: {日期}
```

**存在时**，直接读取，继续 Step 1。

## Step 1：并行数据收集

以下5组搜索**全部同时发出**：

1. `{股票名称} {代码} 2025 2026 券商研究报告 基本面`
2. `{股票名称} {代码} 2026 营收 净利润 业绩 季报`
3. `{股票名称} {代码} 机构评级 目标价 2026`
4. `{股票名称} {代码} 风险 利空 竞争 行业 2026`
5. `{股票名称} {代码} 利好 催化剂 政策 2026`

追加搜索：
- `{股票名称} 股价 主力资金流向 2026年5月`
- `{行业} 退市 ST 失败案例 近5年`（幸存者偏误检查用）

**必须收集**（找不到标注「暂无公开数据」，禁止捏造）：

| 数据项 | 备注 |
|--------|------|
| 当前股价 | 最新收盘价 |
| 机构目标均价 | 近90天共识 |
| 2025全年营收及增速 | YoY |
| 2025归母净利润及增速 | YoY |
| 最新季报营收 | Q1 2026或最新 |
| 核心业务收入 | AI/新能源等新业务 |
| 经营现金流 | 2025年 |
| 机构评级分布 | 买入/增持/中性数量 |
| 近期主力资金流向 | 最近3-5个交易日 |
| 主要财务风险项 | 应收/负债/减值等 |
| 同行退市/ST案例 | 用于幸存者偏误校正 |

## Step 1.5：来源验证（调用 research-triangulation）

**在此处调用 `research-triangulation` 技能**，对 Step 1 收集的数据执行：

- 对每个核心数据点标注来源层级（L1-L6）
- 驱动结论的关键数据确认 ≥2 个独立来源
- 识别所有 L4 卖方研究的潜在利益冲突
- 记录矛盾信息并裁决，L1/L2 之间有矛盾则标红旗

**通过验证才能进入 Step 2。**
无法三角验证的数据降级为「主观判断」档，在后续结论中限制其权重。

---

## Step 2：分析框架

### 持仓参数（用户未指定时使用默认值）
- 持仓金额：20万人民币
- 持仓周期：90天
- 止损比例：-15%

### 多头论证（必涨5条）
每条包含：
- 标题（≤15字）
- 数据依据（2行，具体数字，标注来源层级）
- 强度标签

### 空头论证（必跌5条）
同上格式，使用警示性标签。

### 估值模型（5情景）
**用触发条件代替概率**，每个情景写明进入条件而非拍脑袋的百分比：

| 情景 | 净利润假设 | PE | 目标价 | 进入条件（满足才能触发） |
|------|-----------|-----|--------|------------------------|
| 梦想 | 2028E高点 | 行业高端PE | xx元 | 核心新业务收入占比超15% + 主营同比转正 |
| 乐观 | 2027E | 偏高PE | xx元 | Q2营收降幅收窄至5%内 + 机构资金持续流入 |
| 基准 | 机构共识2026E | 当前PE | xx元 | 机构目标价实现，现状延续 |
| 保守 | 2026E下修 | 中等PE | xx元 | 营收降幅持续扩大 + 关键假设不成立 |
| 悲观 | 继续恶化 | 收缩PE | xx元 | 出现减值公告或重大负面事件 |

当前价在条形图上用竖线标注，max = 最高目标价的110%取整。

### 回报测算（4情景）
- 持股数量 = 持仓金额 / 当前股价
- 4个情景对应触发条件，计算盈亏、EV、风险回报比

## Step 2.5：分析严谨性检查（调用 analytical-rigor）

**在此处调用 `analytical-rigor` 技能**，在输出结论前执行5项检查：

1. **幸存者偏误** — 用 Step 1 搜索到的退市/ST案例校正基准概率
2. **近期外推** — 检查多头/空头理由中是否有不足3个数据点的趋势断言，标注限制
3. **敏感性分析** — 识别整个结论最依赖的单一关键假设，在结论中明确标注
4. **置信度标注** — 对多头/空头每条理由标注：数据证实 / 合理推断 / 主观判断
5. **反向证据** — 主动搜索能推翻主要结论的数据，记录结果

**产出**：一份简洁的「分析质量备注」，附在最终报告的投资结论页底部。

---

## Step 3：Council 四视角辩论（PK评分）

**在此处调用 `council` 技能**，对最终持仓建议进行四视角辩论。

将以下内容传给 council（不传完整对话历史，只传关键事实）：

```
股票：{股票名称} {代码}
核心问题：未来90天，投资者是否应该建仓持有这只股票？

关键数据摘要：
- 营收：{数据} YoY {增速}
- 净利润：{数据} YoY {增速}
- 核心业务：{数据}
- 主力资金：{近期流向}
- 机构评级：{分布}
- 关键假设：{analytical-rigor 识别的最关键假设}
- 幸存者偏误校正后基准概率：{修正结果}
- 已识别的主要风险：{top 3}
```

四个角色的分析焦点：
- **Architect（基本面分析师）**：长期价值、护城河、财务健康度
- **Skeptic（质疑者）**：挑战整个分析框架的前提，问"我们问的是不是错误的问题"
- **Pragmatist（务实者）**：实际投资者能从这个分析中做什么具体操作
- **Critic（风险官）**：下行风险、尾部事件、最坏情况

Council 输出格式：
```
Architect: [立场] [理由] [最大风险]
Skeptic:   [立场] [理由] [最大风险]
Pragmatist:[立场] [理由] [最大风险]
Critic:    [立场] [理由] [最大风险]

共识：[四方一致的部分]
最强反对意见：[最重要的分歧]
前提挑战：[Skeptic 是否挑战了问题本身]
综合裁决：[最终建议]
```

**Council 结果直接用于第4页 PK 评分**，替代单一视角的星级打分。

## Step 4：生成8页 HTML 报告

文件命名：股票名称拼音缩写（茅台→maotai）或股票代码。
输出路径：`{config.repo_path}/{文件名}.html`

**设计规范**：读取 `{config.repo_path}/_DESIGN_SYSTEM.md` 和 `meinian.html`，严格复用。

### 8页结构

| 页 | 内容 | 关键数据来源 |
|----|------|------------|
| 0 | 封面：评级/价格/目标/买点/核心矛盾 | Step 2 |
| 1 | 公司全景：SVG营收图 + 业务矩阵 | Step 1 |
| 2 | 多头：必涨5条 + 置信度标注 | Step 2 + analytical-rigor |
| 3 | 空头：必跌5条 + 置信度标注 | Step 2 + analytical-rigor |
| 4 | Council PK：4视角立场 + 综合裁决 | council 输出 |
| 5 | 估值：5情景触发条件 + 当前价标注 | Step 2 |
| 6 | 结论：评级 + 操作策略 + KPI + 分析质量备注 | analytical-rigor 产出 |
| 7 | 回报：4情景ROI + EV + 风报比 | Step 2 持仓参数 |

第4页 Council PK 组件替换原有星级表，改为展示四个声音的立场卡片 + 综合裁决。

**PresentationController**（键盘/滚轮/触摸/进度条）必须包含。

## Step 5：更新首页 + Git 推送

在 `index.html` 最后一个 `.cards` 区块后追加新报告卡片，包含评级、目标价、页数。

```bash
git -C "{config.repo_path}" add {文件名}.html index.html
git -C "{config.repo_path}" commit -m "feat: add {股票名称} {代码} deep research report"
git -C "{config.repo_path}" push origin main
```

推送成功后返回：
```
✅ 报告已生成并发布

📊 {股票名称} {代码}
🔗 https://{pages_url}/{文件名}.html

分析质量：
  来源验证：{通过/X项待确认}
  关键假设：{analytical-rigor 识别的假设}
  Council裁决：{综合裁决一句话}
  评级：{评级} | 建议买点：{买点} | 止损：{止损价}
```

## 质量检查清单

**输出前必须全部通过：**

- [ ] research-triangulation：核心数据均有 ≥2 个独立来源
- [ ] research-triangulation：无 L6 来源支撑结论
- [ ] analytical-rigor：幸存者偏误已用退市案例校正
- [ ] analytical-rigor：所有趋势外推已标注时间窗口
- [ ] analytical-rigor：关键假设已在报告中明确标注
- [ ] analytical-rigor：置信度档位已标注
- [ ] council：四个视角均已完成，最强反对意见已保留
- [ ] HTML：8页全部存在，设计规范严格复用
- [ ] HTML：第4页展示 Council 输出而非单一星级表
- [ ] HTML：估值页展示触发条件而非裸概率
- [ ] git push：返回成功后才报告完成

## 升级触发：何时跳过 Council

以下情况可跳过 council，直接用 analytical-rigor 的置信度标注：

- 用户明确说「快速分析」或「简版报告」
- 数据极度稀缺（超过5项「暂无公开数据」）

跳过时在报告封面标注「简版·未经Council验证」。
